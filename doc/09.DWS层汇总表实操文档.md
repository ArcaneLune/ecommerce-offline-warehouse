# 电商离线数仓 —— DWS 层汇总表实操文档

> 记录 DWS 层 12 张汇总表从建表到装载的完整流程，基于 Hive on Spark 架构。
> 最后更新：2026-06-26

---

## 一、DWS 层设计概要

| 项目 | 说明 |
|------|------|
| 存储格式 | ORC + Snappy 压缩 |
| 引擎 | Hive on Spark（全链路 Spark 引擎） |
| 数据来源 | DWD 层 10 张事实表 + DIM 层维度表 |
| 分区策略 | 每日一个分区，1d 表按当日分组插入，td 表每日累积 |
| 建表方式 | `CREATE TABLE IF NOT EXISTS`，首次执行后不再重复 |

> DWS 层所有表都从 DWD（Hive ORC 原生表）和 DIM（Hive ORC 原生表）读取，不直接读 Iceberg 源表，因此全程使用 Spark 引擎，无需切 MR。

### 核心设计理念

DWS 层以维度模型的**维度**为基准构建宽表：
- **一行数据** = 一个维度对象当日的汇总行为
- **主键** = 维度 ID（user_id、sku_id、province_id、coupon_id 等）
- **字段构成** = 维度属性（退化维度）+ 事实度量值聚合（次数/件数/金额）
- **目的** = 避免重复计算，提高数据复用性，为 ADS 层提供中间结果

### 统计周期说明

| 后缀 | 含义 | 说明 |
|------|------|------|
| `_1d` | 最近 1 日 | 当日数据独立分区，每日增量运行 |
| `_nd` | 最近 N 日 | 基于 1d 表聚合，计算 7d/30d 滑动窗口 |
| `_td` | 历史至今 | 累积所有历史数据，每日与前一日合并 |

## 三、操作步骤

### 3.1 创建 HDFS 目录

```bash
for t in dws_trade_user_sku_order_1d dws_trade_user_order_1d \
         dws_trade_user_cart_add_1d dws_trade_user_payment_1d \
         dws_trade_province_order_1d \
         dws_traffic_session_page_view_1d dws_traffic_page_visitor_page_view_1d \
         dws_tool_coupon_used_1d dws_interaction_sku_favor_add_1d \
         dws_trade_user_sku_order_nd dws_trade_user_order_td dws_user_user_login_td; do
    hdfs dfs -mkdir -p /warehouse/gmall/dws/${t}
done
```

### 3.2 执行建表 DDL（首次）

```bash
hive -f /opt/software/dws_ddl.sql
```

验证建表：
```bash
hive -e "SHOW TABLES IN gmall" | grep "^dws_"
# 期望输出 12 张表
```

### 3.3 首日装载

```bash
hive --hivevar dt=2026-06-25 -f /opt/software/dws_load_first.sql
```

> 首日装载为全量动态分区。需要 DWD 层和 DIM 层已装载完毕。

### 3.4 每日装载

```bash
hive --hivevar dt=$(date +%Y-%m-%d) -f /opt/software/dws_load_daily.sql
```

> td 表（dws_trade_user_order_td、dws_user_user_login_td）依赖前一天的 td 分区存在。首日用 dws_load_first.sql 初始化，之后每日用 dws_load_daily.sql 累积。

每张表都应有数据。1d 表行数取决于当日活跃维度对象数，td 表行数随历史累积增长。

## 六、关键设计与注意事项

### 6.1 引擎选择

DWS 层所有源表（DWD、DIM）均为 Hive ORC 原生表，不涉及 Iceberg 表直接读取。全程使用 Spark 引擎，无需 `set hive.execution.engine=mr`。

### 6.2 维度退化策略

- **dws_trade_user_sku_order_1d**：通过 LEFT JOIN `dim_sku_full` 退化品类/品牌维度
- **dws_trade_province_order_1d**：通过 LEFT JOIN `dim_province_full` 退化省份/地区维度
- **dws_tool_coupon_used_1d**：通过 LEFT JOIN `dim_coupon_full` 退化优惠券名称/类型
- **dws_trade_user_order_1d / cart_add / payment**：未退化维度，仅保留 user_id，维度信息在 ADS 层按需 JOIN

### 6.3 度量值选取

| 度量类型 | 典型字段 | 聚合方式 |
|------|------|:---:|
| 次数 | order_count、payment_count、coupon_used_count | `COUNT(*)` |
| 去重次数 | order_count（用户粒度订单数） | `COUNT(DISTINCT order_id)` |
| 件数 | order_num、cart_add_num | `SUM(sku_num)` |
| 金额 | order_amount、payment_amount | `SUM(amount_field)` |

### 6.4 td 表累积逻辑

td 表在每日装载时采用 `FULL OUTER JOIN` 前一天分区 + 当天新增：
```
今日 td = 昨日 td (dt=day-1) FULL OUTER JOIN 今日 1d (dt=today)
```
新增用户初始化为当天数据，存量用户各指标累加。首次装载时不存在昨日分区，全部由首日 SQL 初始化。

### 6.5 nd 表窗口计算

7d/30d 窗口通过 `IF(dt >= date_add(基准日期, -N+1), 指标, 0)` 条件求和实现：
- 首日：对 dws_trade_user_sku_order_1d 全表聚合，WHERE 不过滤，IF 条件内取最近 N 日
- 每日：WHERE dt >= date_add('${hivevar:dt}', -29)，确保只扫描窗口内分区

### 6.6 命名规范

`dws_{数据域}_{统计粒度}_{业务过程}_{统计周期}`
- 数据域：trade / traffic / tool / interaction / user
- 统计粒度：user、user_sku、province、session、page_visitor、coupon、sku
- 统计周期：1d / nd / td

## 七、与旧项目差异

| 旧项目（Maxwell CDC / old Spark） | 新架构（Hive on Spark） |
|----------------------|----------------------------|
| `DROP TABLE IF EXISTS` | `CREATE TABLE IF NOT EXISTS` |
| `nvl(col, 0)` | `COALESCE(col, CAST(0 AS ...))` |
| 硬编码日期 `'2022-06-08'` | `${hivevar:dt}` 变量 |
| `set hive.vectorized.execution.enabled = false` | 无需设置（全程 Spark 引擎） |
| 引擎混用（Iceberg 切 MR） | 全程 Spark（DWS 不读 Iceberg） |
| `dws_user_user_login_td` 缺 `login_date_first` | 已补齐（ADS 留存率统计需要） |
| `dws_trade_province_order_1d` 缺 `iso_3166_2` | 已补齐（ADS 省份地图可视化需要） |
