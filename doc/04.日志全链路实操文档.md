## 一、日志链路总览

### 1.1 数据流向

```
日志模拟生成器      Filebeat 7.17.24       Kafka 3.6.2          SeaTunnel 2.3.13        HDFS                    Hive/Spark
(hadoop101)         (hadoop101)            (三节点集群)          (hadoop101)             (HDFS集群)              (hadoop100)
    ↓                   ↓                    ↓                    ↓                       ↓                           ↓
生成 .log 文件     tail 读取 + JSON 校验    topic_log            流式消费 + 按日分区     /warehouse/gmall/        JsonSerDe
/data/logs/app/    Kafka 推送              (3分区, 1副本)        写入 JSON(.gz)         ods/ods_log_inc/        EXTERNAL TABLE
                                                                                      dt=YYYY-MM-DD/           ods_log_inc
```

### 1.2 关键组件

| 组件 | 版本 | 节点 | 角色 |
|------|------|:---:|------|
| 日志模拟生成器 | Python/Shell | hadoop101 | 模拟用户行为，产生 .log 文件 |
| Filebeat | 7.17.24 | hadoop101 | tail 文件 + JSON 格式校验 + 发送 Kafka |
| Kafka | 3.6.2 | 三节点 | topic_log 消息队列中转（3分区1副本，保留72h） |
| SeaTunnel | 2.3.13 | hadoop101 | 流式消费 Kafka → 按日分区写入 HDFS |
| HDFS | 3.3.6 | 三节点 | 文件存储 |
| Hive Metastore | 3.1.3 | hadoop100 | 元数据管理 |

### 1.3 核心约束

**ODS 层建表语句是整个日志链路的唯一目标。** 无论上游（日志生成、Filebeat、SeaTunnel）如何实现，最终写入 HDFS 的数据必须能被以下 DDL 正确读取：

```sql
DROP TABLE IF EXISTS ods_log_inc;
CREATE EXTERNAL TABLE ods_log_inc
(
    `common` STRUCT<ar :STRING,
        ba :STRING,
        ch :STRING,
        is_new :STRING,
        md :STRING,
        mid :STRING,
        os :STRING,
        sid :STRING,
        uid :STRING,
        vc :STRING> COMMENT '公共信息',
    `page` STRUCT<during_time :STRING,
        item :STRING,
        item_type :STRING,
        last_page_id :STRING,
        page_id :STRING,
        from_pos_id :STRING,
        from_pos_seq :STRING,
        refer_id :STRING> COMMENT '页面信息',
    `actions` ARRAY<STRUCT<action_id:STRING,
        item:STRING,
        item_type:STRING,
        ts:BIGINT>> COMMENT '动作信息',
    `displays` ARRAY<STRUCT<display_type :STRING,
        item :STRING,
        item_type :STRING,
        `pos_seq` :STRING,
        pos_id :STRING>> COMMENT '曝光信息',
    `start` STRUCT<entry :STRING,
        first_open :BIGINT,
        loading_time :BIGINT,
        open_ad_id :BIGINT,
        open_ad_ms :BIGINT,
        open_ad_skip_ms :BIGINT> COMMENT '启动信息',
    `err` STRUCT<error_code:BIGINT,
            msg:STRING> COMMENT '错误信息',
    `ts` BIGINT  COMMENT '时间戳'
) COMMENT '活动信息表'
    PARTITIONED BY (`dt` STRING)
    ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.JsonSerDe'
LOCATION '/warehouse/gmall/ods/ods_log_inc/'
TBLPROPERTIES ('compression.codec'='org.apache.hadoop.io.compress.GzipCodec');
```

```json
        "ba": "iPhone",

        "md": "iPhone 8",

        "os": "iOS 13.2.9",
        "sid": "3981c171-558a-437c-be10-da6d2553c517",
        "uid": "485",
        "vc": "v2.1.134"
    },
    "actions": [{
        "action_id": "favor_add",
        "item": "3",
        "item_type": "sku_id",
        "ts": 1585744376605
    }],
    "displays": [{
        "display_type": "query",
        "item": "3",
        "item_type": "sku_id",
        "pos_seq": "1",
        "pos_id": "2"
    }],
    "page": {
        "during_time": "7648",
        "item": "3",
        "item_type": "sku_id",
        "last_page_id": "login",
        "page_id": "good_detail",
        "from_pos_id": "999",
        "from_pos_seq": "999",
        "refer_id": "2"
    },
    "err": {
        "error_code": "1234",
        "msg": "***********"
    },
    "ts": 1585744374423
}
```

#### 启动日志

以启动为单位——一次启动行为包含：环境信息(`common`)、启动记录(`start`)、一个报错记录(`err`)和时间戳(`ts`)。

```json
{
    "common": {
        "ar": "370000",
        "ba": "Honor",
        "ch": "wandoujia",
        "is_new": "1",
        "md": "Honor 20s",
        "mid": "eQF5boERMJFOujcp",
        "os": "Android 11.0",
        "sid": "a1068e7a-e25b-45dc-9b9a-5a55ae83fc81",
        "uid": "76",
        "vc": "v2.1.134"
    },
    "start": {
        "entry": "icon",
        "first_open": 0,
        "loading_time": 18803,
        "open_ad_id": 7,
        "open_ad_ms": 3449,
        "open_ad_skip_ms": 1989
    },
    "err": {
        "error_code": "1234",
        "msg": "***********"
    },
    "ts": 1585744304000
}
```

> **必须与 ODS 建表 DDL 对齐的字段名（JsonSerDe 区分大小写）**：
> - `common` 的 10 个子字段全部用小写
> - `actions` 数组内每个对象的字段：`action_id`, `item`, `item_type`, `ts`
> - `displays` 数组内每个对象的字段：`display_type`, `item`, `item_type`, `pos_seq`, `pos_id`
> - `page` 的 8 个子字段：`during_time`, `item`, `item_type`, `last_page_id`, `page_id`, `from_pos_id`, `from_pos_seq`, `refer_id`
> - `start` 的 6 个子字段：`entry`, `first_open`, `loading_time`, `open_ad_id`, `open_ad_ms`, `open_ad_skip_ms`
> - `err` 的 2 个子字段：`error_code`, `msg`
> - `ts` 顶层字段

### 1.5 两类日志在 ODS 表中的共存

两类日志写入同一个 `.log` 文件 → Filebeat 逐行采集为独立 Kafka 消息 → SeaTunnel 写入同一 HDFS 目录 → Hive JsonSerDe 读取为同一张表：

| 字段 | 页面日志 | 启动日志 |
|------|:---:|:---:|
| `common` | ✅ 有值 | ✅ 有值 |
| `page` | ✅ 有值 | NULL |
| `actions` | ✅ 有值 | NULL |
| `displays` | ✅ 有值 | NULL |
| `start` | NULL | ✅ 有值 |
| `err` | ✅ 有值 | ✅ 有值 |
| `ts` | ✅ 有值 | ✅ 有值 |

## 四、第三步：Filebeat 采集 + Kafka 验证

**执行节点**：hadoop101（Filebeat 所在节点）、三节点（Kafka 集群）
**目标**：Filebeat 实时采集 `/data/logs/app/*.log`，JSON 校验后发送到 Kafka topic_log，验证全链路无丢失。

---

### 4.1 业务背景

日志模拟器持续向 `/data/logs/app/app_YYYY-MM-DD.log` 写入两类日志（页面日志 + 启动日志），每行一条独立 JSON。Filebeat 的角色：

1. **tail -f** 等效：监控文件增量，新写入的行实时采集
2. **JSON 校验**：合法 JSON → 发送 Kafka；不合法 JSON → 丢弃并标记 `error.message`
3. **解耦**：日志生产者和消费者互不依赖，Kafka 作为中间缓冲

### 4.4 Filebeat 配置

```bash
vim /opt/module/filebeat-7.17.24/filebeat.yml
```

完整配置（覆盖原文件）：

```yaml
# ======================== 输入 ========================
filebeat.inputs:
- type: log
  enabled: true
  paths:
    - /data/logs/app/*.log

  # ===== JSON 解析配置 =====
  # 将 JSON 字段提升为 Kafka 消息的顶层字段（而非嵌套在 "message" 里）
  json.keys_under_root: true
  # 如果消息本身已有字段，JSON 内容覆盖它
  json.overwrite_keys: true
  # 解析失败时在消息中追加 error.message 字段（便于排查），但本条消息不进 Kafka
  json.add_error_key: true
  # 不向 Kafka 发送解析失败的消息
  json.ignore_decoding_error: true

  # 标记来源
  tags: ["user_behavior"]

# ======================== 处理：发送前剥掉 Filebeat 元数据 ========================
processors:
  - drop_fields:
      fields: ["agent", "host", "ecs", "input", "log", "tags"]
      ignore_missing: true
  # 注意：@timestamp 和 @metadata 是 Filebeat 内部字段，drop_fields 无法删除。
  # 它们会保留在 Kafka 消息中，Spark 的 JsonSerDe 读取时自动忽略不认识的字段。

# ======================== 输出 ========================
output.kafka:
  # 三节点 Broker 列表——Filebeat 自动发现并负载均衡
  hosts: ["hadoop100:9092", "hadoop101:9092", "hadoop102:9092"]
  topic: "topic_log"
  # 轮询分发到各分区
  partition.round_robin:
    reachable_only: false
  # leader 确认即返回（学习集群 1 副本，无需等 ISR）
  required_acks: 1
  # gzip 压缩：减少 70-80% 网络带宽
  compression: gzip
  # 单条消息上限（超过此值的消息直接丢弃）
  max_message_bytes: 1000000
```

**配置逐条解读（为什么这样设）**：

| 配置项 | 值 | 业务原因 |
|------|------|------|
| `json.keys_under_root` | true | 如果不设，整个 JSON 会变成 `{"message": "{...}"}` 嵌套；设为 true 后 `common`、`page`、`ts` 等变成顶层，下游 SeaTunnel 直接消费原始结构 |
| `json.overwrite_keys` | true | 如果 Filebeat 自动添加的字段（如 `@timestamp`）和 JSON 内的字段同名，以 JSON 为准 |
| `json.add_error_key` + `json.ignore_decoding_error` | true + true | 两段式处理：先标记错误，再丢弃——既保留排查线索，又不让脏数据污染下游 |
| `compression: gzip` | gzip | 一条页面日志约 500-2000 字节（含 actions/displays 数组），10000 条约 15MB，gzip 后约 3-4MB |
| `required_acks: 1` | 1 | 学习环境 1 副本，acks=all 无意义；acks=1 保证 leader 写入成功 |
| `max_message_bytes` | 1MB | 单条日志极少超过 100KB，1MB 上限是安全网 |

### 4.5 配置 Systemd 服务（后台运行）

Filebeat 通过 systemd 管理，开机自启、异常退出自动重启，比 nohup 更稳定。

##### 1. 创建服务文件

```bash
sudo vim /etc/systemd/system/filebeat.service
```

```ini
[Unit]
Description=Filebeat - User Behavior Log Collector
Documentation=https://www.elastic.co/guide/en/beats/filebeat/current/index.html
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=hadoop
Group=hadoop
WorkingDirectory=/opt/module/filebeat-7.17.24
ExecStartPre=/opt/module/filebeat-7.17.24/filebeat test config
ExecStart=/opt/module/filebeat-7.17.24/filebeat -e
Restart=on-failure
RestartSec=10
TimeoutStopSec=10
MemoryMax=256M

[Install]
WantedBy=multi-user.target
```

##### 2. 启动与管理

```bash
# 重载 systemd 配置
sudo systemctl daemon-reload

# 前台测试：确认能连接到 Kafka 且无 ERROR
cd /opt/module/filebeat-7.17.24/
./filebeat -e
# Ctrl+C 退出

# 通过 systemd 正式启动
sudo systemctl start filebeat
sudo systemctl enable filebeat   # 开机自启

# 查看状态
sudo systemctl status filebeat

# 后续管理
sudo systemctl stop filebeat      # 停止
sudo systemctl restart filebeat   # 重启
sudo systemctl disable filebeat   # 取消开机自启
```

##### 3. 日志位置

| 日志类型 | 位置 | 说明 |
|------|------|------|
| Filebeat 运行日志 | `/opt/module/filebeat-7.17.24/logs/filebeat` | 滚动日志文件，包含采集详情和 Kafka 连接状态 |
| Filebeat JSON 解析错误 | `/opt/module/filebeat-7.17.24/logs/filebeat` | 含 `error.message` 的条目在此可见 |
| Systemd 服务日志 | `sudo journalctl -u filebeat` | systemd 管理的 stdout/stderr 输出 |

```bash
# 查看 Filebeat 自身日志
tail -50 /opt/module/filebeat-7.17.24/logs/filebeat

# 查看最近的 systemd 日志
sudo journalctl -u filebeat -n 30 --no-pager

# 如果 Filebeat 起不来，优先看 systemd 日志
sudo journalctl -u filebeat -n 20 --no-pager | grep -i error
```

### 5.3 作业配置文件

```bash

vim /opt/module/seatunnel-2.3.13/config/jobs/kafka_log_to_hdfs.conf
```

```hocon
env {
  job.mode = "STREAMING"
  parallelism = 1
  checkpoint.interval = 60000
}

source {
  Kafka {
    topic = "topic_log"
    bootstrap.servers = "hadoop100:9092,hadoop101:9092,hadoop102:9092"
    consumer.group = "seatunnel_log_consumer"
    format = json
    start_mode = "group_offsets"
    plugin_output = "kafka_source"
  }
}

# === Transform：生成分区字段 dt，只取 content 列 ===
# SeaTunnel Kafka source 的 format=json 会把整条 Kafka 消息作为字符串放入 content 列
# 这里用 SQL 取出 content，并基于 current_date 生成 dt 分区键
transform {
  Sql {
    plugin_input = "kafka_source"
    query = "select content, cast(current_date as varchar) as dt from kafka_source"
    plugin_output = "kafka_with_dt"
  }
}

# === Sink：text 格式原样写入（不做 JSON 二次包装） ===
sink {
  HdfsFile {
    plugin_input = "kafka_with_dt"
    fs.defaultFS = "hdfs://hadoop100:8020"
    path = "/warehouse/gmall/ods/ods_log_inc"
    # text 格式：每列值原样写入文件，用 row_delimiter 分隔行
    # json 格式会导致 SeaTunnel 用 {"content": "..."} 二次包装，JsonSerDe 无法解析
    file_format_type = "text"
    field_delimiter = ""
    row_delimiter = "\n"
    have_partition = true
    partition_by = ["dt"]
    partition_dir_expression = "${k0}=${v0}"
    # dt 仅用于分区路径计算，不写入文件（避免数据列和分区列重名）
    is_partition_field_write_in_file = false
    filename_time_format = "HHmmss"
    batch_size = 1000000
    is_enable_transaction = true
  }
}
```

**配置与 ODS 表的逐项对照**：

| 配置行 | 值 | 对应 ODS 表的什么 |
|------|------|------|
| `path` | `/warehouse/gmall/ods/ods_log_inc` | `LOCATION '/warehouse/gmall/ods/ods_log_inc/'` |
| `partition_dir_expression` | `dt=${now}` | `PARTITIONED BY (dt STRING)`，每天生成 `dt=2026-06-18/` 目录 |
| `file_format` / `file_format_type` | `json` / `json` | `ROW FORMAT SERDE 'JsonSerDe'`，每条消息即一行 JSON |
| `is_enable_transaction` | `true` | 防丢防重，重启后从 checkpoint 恢复 |
| 无 `compress_codec` | 不配置 | SeaTunnel 2.3.13 不支持 gzip。写入的文件为 `.json`，Spark JsonSerDe 同时兼容压缩/非压缩文件，读取不受影响 |

## 七、差异分析：原始设计 vs 实际实现

### 7.1 组件层面

| 组件 | 原始设计 | 实际实现 | 变更原因 |
|------|---------|---------|---------|
| Filebeat | 不加 `processors`，元数据字段随日志一起入 Kafka | 加了 `processors.drop_fields` 去掉 `agent/host/ecs/input/log/tags` | 元数据字段干扰数据观测；`@timestamp` 和 `@metadata` 无法删除但 JsonSerDe 自动忽略 |
| SeaTunnel 写入格式 | `file_format_type = "json"` | `file_format_type = "text"` | JSON 格式导致 SeaTunnel 把行数据包装为 `{"content": "..."}`，JsonSerDe 无法解析嵌套在 content 内的真实字段 |
| SeaTunnel 分区方式 | `partition_dir_expression = "dt=${now}"` | transform Sql 生成 `dt` 列 + `partition_by = ["dt"]` + `${k0}=${v0}` | SeaTunnel 2.3.13 要求 `partition_by` 为数组类型；`dt=${now}` 语法在该版本不支持 |
| SeaTunnel 内存参数 | `max_rows_in_memory` | `batch_size` | SeaTunnel 2.3.13 改名 |
| ODS 表压缩 | `TBLPROPERTIES (...GzipCodec)` | 保留未改 | SeaTunnel 不支持 gzip 写入，但 TBLPROPERTIES 只控制 Hive 写入不控制 Spark 读取 |
| Spark JAR 依赖 | 未提及 | 建表时需要 `hive-serde` + `hive-common` 放入 `$SPARK_HOME/jars/`；查询时删除 `hive-exec`（与 Spark 内置版本冲突） | Spark on Hive 模式下 Spark 自己维护 Hive 客户端，版本不一致导致 NoSuchMethodError |

### 7.2 数据流层面

| 环节 | 原始设计 | 实际实现 | 差异 |
|------|---------|---------|------|
| Kafka 消息 | 纯日志 JSON | 日志 JSON + Filebeat 元数据（`@timestamp`、`@metadata`） | `processors` 可删大部分元数据，`@timestamp/@metadata` 是硬编码无法删除 |
| SeaTunnel 读取 | JSON 字段解析为独立列 | 整条消息作为字符串放入 `content` 列 | Kafka source 的 `format=json` 不展开嵌套结构 |
| HDFS 文件内容 | 每行一个纯 JSON 对象 | 同设计 | `text` 格式 + `field_delimiter = ""` 实现了与设计一致的效果 |
| HDFS 文件名 | `.json.gz` | `.txt` | `file_format_type = "text"` 默认给 `.txt` 后缀；Spark 读取不依赖后缀 |
| 分区字段 | SeaTunnel 直接写 `dt=${now}` | transform Sql 从时间戳生成 dt 列 | 原始数据中没有 dt 字段，需要 ETL 过程中计算 |
