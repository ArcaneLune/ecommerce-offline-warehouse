# 电商离线数仓 —— 全量表业务逻辑与数据字典

> 覆盖 MySQL 源库 29 张 + 数仓 76 张，共 105 张表。逐表说明业务含义、数据粒度、核心字段和上下游关系。

---

## 第一部分：MySQL 业务源库（gmall，29 张表）

MySQL 模拟一个电商平台的业务数据库，涵盖商品、订单、用户、营销、购物车五大业务域。

### 一、商品域（7 张）

商品的 SPU → SKU 层次结构，以及品牌、品类、属性。

| 表名 | 业务含义 | 一行代表 | 核心字段 |
|------|---------|---------|---------|
| `sku_info` | SKU 商品主表，库存量单位 | 一个具体 SKU（如"iPhone 15 黑色 256GB"） | id, spu_id, price, sku_name, tm_id, category3_id, is_sale |
| `spu_info` | SPU 标准产品单位 | 一个产品系列（如"iPhone 15"） | id, spu_name, category3_id, tm_id |
| `base_trademark` | 品牌维度表 | 一个品牌（如 Apple、华为） | id, tm_name |
| `base_category1` | 一级品类 | 一个大类（如 3C 数码） | id, name |
| `base_category2` | 二级品类 | 一个中类（如 手机） | id, name, category1_id |
| `base_category3` | 三级品类 | 一个小类（如 智能手机） | id, name, category2_id |
| `sku_attr_value` | SKU 平台属性值（规格参数） | 一个 SKU 的一个属性（如"屏幕尺寸:6.1英寸"） | sku_id, attr_id, value_id, attr_name, value_name |
| `sku_sale_attr_value` | SKU 销售属性值（颜色/容量等） | 一个 SKU 的一个销售属性（如"颜色:黑色"） | sku_id, spu_id, sale_attr_value_name |

**业务关系：** `spu_info → sku_info → sku_attr_value / sku_sale_attr_value`，品类三级树形 `category1 → category2 → category3`。

### 二、订单域（7 张）

电商核心交易链路：下单 → 支付 → 发货确认，每个环节有对应的流水记录。

| 表名 | 业务含义 | 一行代表 | 核心字段 |
|------|---------|---------|---------|
| `order_info` | 订单主表 | 一笔订单 | id, user_id, total_amount, order_status, province_id, create_time |
| `order_detail` | 订单明细表 | 一笔订单中的一个商品项 | id, order_id, sku_id, sku_num, order_price, split_total_amount |
| `order_detail_activity` | 订单明细参与活动记录 | 一个订单商品项参与的一项活动 | order_detail_id, activity_id, activity_rule_id |
| `order_detail_coupon` | 订单明细使用优惠券记录 | 一个订单商品项使用的一张券 | order_detail_id, coupon_id |
| `order_status_log` | 订单状态变更流水 | 一次状态变更（如"已支付→已发货"） | order_id, order_status, create_time |
| `payment_info` | 支付流水表 | 一笔支付记录 | order_id, payment_type, total_amount, callback_time, payment_status |
| `order_refund_info` | 退款/退货表 | 一笔退款申请 | order_id, sku_id, refund_amount, refund_status |
| `refund_payment` | 退款支付流水 | 一笔退款打款记录 | order_id, refund_status, callback_time |

**核心状态码：** `order_status`: 1001待支付→1002待发货→1003待收货→1004已收货；`payment_status`: 1601未支付→1602已支付→1603已退款。

### 三、用户与互动域（4 张）

| 表名 | 业务含义 | 一行代表 | 核心字段 |
|------|---------|---------|---------|
| `user_info` | 用户主表 | 一个注册用户 | id, name, phone_num, email, user_level, birthday, gender |
| `favor_info` | 商品收藏表 | 一次收藏行为 | user_id, sku_id, is_cancel, create_time |
| `comment_info` | 商品评价表 | 一条评价 | user_id, sku_id, order_id, appraise(好评/中评/差评) |
| `coupon_use` | 优惠券领用表 | 一张券的领取→使用全生命周期 | coupon_id, user_id, order_id, coupon_status, used_time |

### 四、营销域（4 张）

| 表名 | 业务含义 | 一行代表 | 核心字段 |
|------|---------|---------|---------|
| `coupon_info` | 优惠券定义表 | 一种优惠券 | coupon_name, coupon_type(现金/折扣/满减), benefit_amount, condition_amount |
| `activity_info` | 促销活动定义表 | 一个活动（如双十一满减） | activity_name, activity_type(满减/折扣), start_time, end_time |
| `activity_rule` | 活动规则表 | 活动的一条优惠规则（如"满300减50"） | activity_id, condition_amount, benefit_amount |
| `promotion_pos` | 营销坑位表 | 一个 App 内的广告位 | pos_location, pos_type |
| `promotion_refer` | 营销渠道表 | 一个外部推广渠道（如"抖音广告"） | refer_name |

### 五、购物车域（1 张）

| 表名 | 业务含义 | 一行代表 | 核心字段 |
|------|---------|---------|---------|
| `cart_info` | 购物车表 | 一个用户购物车中的一个商品 | user_id, sku_id, sku_num, is_ordered, create_time |

### 六、公共域（2 张）

| 表名 | 业务含义 | 一行代表 |
|------|---------|---------|
| `base_dic` | 字典表 | 一个编码及其含义（如 payment_type: 1101=微信, 1102=支付宝） |
| `base_province` | 省份表 | 一个省份（含 ISO 地区编码） |
| `base_region` | 大区表 | 一个大区（如 华东、华北） |

## 第二部分：ODS 层（30 张表）

ODS 层保持与源系统一致，不做业务加工。分为三类：

### 全量同步表（16 张 Hive TEXT 表，每日覆盖一个分区）

同步策略：每日凌晨 DataX 全量拉取，写入 `dt=YYYY-MM-DD` 分区，覆盖式写入。

**数据源对应关系：** 16 张表与 MySQL 中维度/字典/低频变更表一一对应：
`activity_info, activity_rule, base_category1/2/3, base_dic, base_province, base_region, base_trademark, coupon_info, promotion_pos, promotion_refer, sku_attr_value, sku_info, sku_sale_attr_value, spu_info`

> 这些表的特点是数据量小、变更频率低（如品类树基本不变、品牌列表按月更新），全量覆盖即可满足分析需求。

### 增量同步表（13 张 Iceberg v2 表，Flink CDC 持续 Upsert）

同步策略：Flink CDC 实时追 MySQL binlog，经过 Kafka 中继写入 Iceberg。每张表按 `dt`（从 create_time/operate_time 提取）分区。

**为什么这 13 张表用增量：** 它们记录的是业务流程中的事实数据（订单、支付、用户行为），数据量大且持续产生，增量同步保证实时性和数据完整性。

| 表名 | 业务含义 | 为什么增量 |
|------|---------|-----------|
| `ods_order_info_inc` | 订单主表 | 实时下单，每日数万笔 |
| `ods_order_detail_inc` | 订单明细 | 与订单 1:N，数据量更大 |
| `ods_order_detail_activity_inc` | 订单参与活动 | 与明细关联 |
| `ods_order_detail_coupon_inc` | 订单使用优惠券 | 与明细关联 |
| `ods_order_status_log_inc` | 订单状态流水 | 持续产生状态变更 |
| `ods_payment_info_inc` | 支付流水 | 支付是核心业务事件 |
| `ods_refund_payment_inc` | 退款流水 | 退款需要增量追踪 |
| `ods_order_refund_info_inc` | 退款申请 | 退款状态会变更 |
| `ods_user_info_inc` | 用户信息 | 用户注册、资料修改频繁 |
| `ods_favor_info_inc` | 收藏行为 | 高频用户行为 |
| `ods_coupon_use_inc` | 优惠券领用 | 领券+使用持续发生 |
| `ods_comment_info_inc` | 商品评价 | 用户持续评价 |
| `ods_cart_info_inc` | 购物车 | 加购/减购频繁变更 |

### 日志表（1 张 Hive JSON SerDe 表）

| 表名 | 业务含义 | 一行代表 |
|------|---------|---------|
| `ods_log_inc` | 用户行为日志 | 一次页面浏览或一次 App 启动 |

包含环境信息（设备、渠道、版本）、页面信息（page_id, during_time）、动作数组（点击/加购/收藏）、曝光数组、错误信息。由 Filebeat→Kafka→SeaTunnel 流式写入。

## 第三部分：DIM 层（8 张维度表）

DIM 层为 DWD 事实表提供维度信息，回答"谁、什么、何时、何地"的描述性问题。

### 3.1 dim_sku_full（商品维度表）—— 每日全量快照

**一行代表：** 一个 SKU 的完整信息（含品牌、品类、属性、SPU）。

**数据来源：** 8 张 ODS 全量表的 JOIN：`sku_info + spu_info + base_trademark + base_category1/2/3 + sku_attr_value + sku_sale_attr_value`

**核心字段：** sku_name, price, spu_name, tm_name, category1/2/3_name, sku_attr_values(平台属性数组), sku_sale_attr_values(销售属性数组)

**业务作用：** 为 DWD 订单明细表和 DWS 汇总表提供商品的品类路径和品牌信息，实现维度退化。在 ADS 层的品牌复购率、品类下单统计中直接使用。

### 3.2 dim_coupon_full（优惠券维度表）—— 每日全量快照

**一行代表：** 一张优惠券的完整定义信息。

**核心字段：** coupon_name, coupon_type_name(现金券/折扣券/满减券/满件打折), condition_amount, benefit_amount, benefit_rule

**业务作用：** 在 ADS 优惠券使用统计中提供券的名称和类型。

### 3.3 dim_activity_full（活动维度表）—— 每日全量快照

**一行代表：** 一条活动规则（活动 ID + 规则 ID 的组合）。

**核心字段：** activity_name, activity_type_name, condition_amount, benefit_amount, start_time, end_time

**业务作用：** 分析不同促销活动的效果（转化率、带来的 GMV）。

### 3.4 dim_province_full（地区维度表）—— 每日全量快照

**一行代表：** 一个省份及其所属大区。

**核心字段：** province_name, area_code, iso_code, iso_3166_2, region_name

**业务作用：** ADS 省份交易统计、Superset 中国地图可视化（需要 ISO 地区编码做地理映射）。

### 3.5 dim_user_zip（用户拉链表）—— 缓慢变化维

**一行代表：** 一个用户在某个时间段内的状态版本。

**核心字段：** user_id, name(脱敏), phone_num(脱敏), email(脱敏), user_level, birthday, gender, start_date, end_date

**拉链逻辑：**
- 首日：所有用户以当前状态写入 `dt=9999-12-31` 分区，`end_date='9999-12-31'`
- 每日：ods_user_info_inc 中有变更的用户 → 关旧链（旧记录 end_date 设为昨天）+ 开新链（新记录 start_date=今天, end_date='9999-12-31'）
- 查询当前有效用户：`WHERE end_date='9999-12-31'`
- 查询历史快照：`WHERE start_date<='2026-06-01' AND end_date>'2026-06-01'`

**业务作用：** 支持用户画像随时间变化的分析，如"去年这个时候用户等级分布是怎样的"。

### 3.6-3.8 辅助维度表

| 表名 | 含义 | 特点 |
|------|------|------|
| `dim_promotion_pos_full` | 营销坑位维度 | 提供广告位名称，用于流量域分析 |
| `dim_promotion_refer_full` | 营销渠道维度 | 提供渠道名称，用于渠道转化分析 |
| `dim_date` | 日期维度（一次性导入，1095 行） | date_id, week_id, week_day, month, quarter, year, is_workday, holiday_id。覆盖 2025-2027 |

## 第四部分：DWD 层（10 张明细事实表）

DWD 层是数仓的核心明细层，每张表对应一个具体的业务过程，一行代表一次不可再分的业务原子事件。

### 4.1 dwd_trade_cart_add_inc（加购事务事实表）

**业务过程：** 用户将商品加入购物车。

**一行代表：** 一次加购行为。

**数据来源：** ods_cart_info_inc（Iceberg 增量表，通过 MR 引擎读取）

**核心字段：** user_id, sku_id, sku_num, create_time

**下游使用：** 统计加购人数、加购商品数（用户行为漏斗分析中的一环）。

### 4.2 dwd_trade_order_detail_inc（下单事务事实表）

**业务过程：** 用户提交订单（一笔订单中的每个商品一行）。

**一行代表：** 一笔订单中的一个商品项。

**数据来源：** ods_order_detail_inc + ods_order_info_inc（获取 user_id, province_id）+ ods_order_detail_activity_inc（获取活动信息）+ ods_order_detail_coupon_inc（获取优惠券信息）

**核心字段：**

| 字段 | 业务含义 |
|------|---------|
| order_id | 订单 ID，同一订单多个商品时此字段重复 |
| user_id | 下单用户 |
| sku_id | 购买的商品 |
| sku_num | 购买数量 |
| split_original_amount | 该商品项原始金额（单价×数量） |
| split_activity_amount | 该商品项分摊的活动优惠金额 |
| split_coupon_amount | 该商品项分摊的优惠券优惠金额 |
| split_total_amount | 该商品项实付金额（original - activity - coupon） |
| province_id | 收货省份（已退化到事实表中） |
| activity_id / coupon_id | 参与的活动/使用的优惠券（已退化） |

**维度退化设计：** 订单表 JOIN 活动表和优惠券表后，将 activity_id、coupon_id 直接沉淀到 DWD 宽表中，避免下游每次分析都要重复 JOIN。

**下游使用：** 这是数仓最重要的一张事实表。DWS 层 4 张交易域汇总表和 ADS 层 6 张报表均直接或间接依赖此表。

### 4.3 dwd_trade_pay_detail_suc_inc（支付成功事务事实表）

**业务过程：** 用户完成支付（仅包含支付成功的记录）。

**一行代表：** 一笔支付成功的订单商品项。

**数据来源：** ods_order_detail_inc + ods_payment_info_inc（WHERE payment_status='1602'，仅取支付成功）

**与 4.2 的关键区别：**
- 4.2 按下单时间分区，包含所有下单（含未支付的）
- 4.3 按支付回调时间分区，仅含已支付的
- 4.3 多了 payment_type 字段

**业务作用：** 统计实际收入（GMV 以支付为准），分析不同支付方式的占比。

### 4.4 dwd_trade_cart_full（购物车周期快照事实表）

**业务过程：** 每日截止时刻购物车中仍存在的商品（未下单的）。

**一行代表：** 一个用户的购物车中一个未下单的商品项。

**数据来源：** ods_cart_info_full（Hive TEXT 全量表，每日快照）

**业务作用：** 统计各品类购物车存量 TopN、分析用户购物车放弃率。

### 4.5 dwd_trade_trade_flow_acc（交易流程累积快照事实表）

**业务过程：** 一笔订单从下单到支付到确认收货的完整生命周期。

**一行代表：** 一笔订单的完整交易流程（三个里程碑时间）。

**核心字段：** order_id, order_time, payment_time, finish_time（确认收货时间）, order_original_amount, payment_amount

**设计意义：** 将多个业务过程（下单、支付、确认收货）合并到一张表，避免分析"下单到支付耗时"时需要 JOIN 三张表。未完成的流程保存在 `dt='9999-12-31'` 分区中，完成后移动到确认收货日期分区。

**业务作用：** 计算下单到支付的平均时间间隔（ADS 层指标）。

### 4.6 dwd_tool_coupon_used_inc（优惠券使用事务事实表）

**业务过程：** 用户使用优惠券完成支付。

**一行代表：** 一张优惠券的一次使用记录。

**数据来源：** ods_coupon_use_inc（WHERE used_time IS NOT NULL）

**业务作用：** 统计优惠券使用次数、使用人数，评估优惠券活动的效果。

### 4.7 dwd_interaction_favor_add_inc（收藏商品事务事实表）

**业务过程：** 用户收藏一个商品。

**一行代表：** 一次收藏行为。

**数据来源：** ods_favor_info_inc

**业务作用：** 统计各商品被收藏次数、各品牌被收藏 TopN。

### 4.8 dwd_traffic_page_view_inc（页面浏览事务事实表）

**业务过程：** 用户在 App/网站中浏览一个页面。

**一行代表：** 一次页面浏览（一次页面跳入）。

**数据来源：** ods_log_inc（日志 JSON 表，WHERE page IS NOT NULL）

**字段展开：** 日志中 `common` 对象的每个字段（ar→province_id, ba→brand, ch→channel 等）和 `page` 对象的每个字段（page_id, during_time, item 等）被展开为独立的列。`ts` 毫秒时间戳转换为北京时间字符串。

**业务作用：** 流量分析的基础表。统计各页面 PV/UV、各渠道流量、页面跳转路径分析、会话分析。

### 4.9 dwd_user_register_inc（用户注册事务事实表）

**业务过程：** 用户完成注册。

**一行代表：** 一个用户的注册事件。

**数据来源：** ods_user_info_inc + ods_log_inc（关联注册页面的日志获取渠道、设备信息）

**业务作用：** 统计每日新增用户数、分析各渠道的新用户质量。

### 4.10 dwd_user_login_inc（用户登录事务事实表）

**业务过程：** 用户登录 App。

**一行代表：** 一个用户会话的首次页面浏览（登录行为）。

**数据来源：** ods_log_inc（每个 session 取 ROW_NUMBER=1 的第一条）

**业务作用：** 统计 DAU/WAU/MAU（日活/周活/月活）、分析用户留存和回流。

## 第五部分：DWS 层（12 张汇总表）

DWS 层对 DWD 明细数据做轻度聚合，以维度对象为粒度，统计当日/近 N 日/历史至今的度量值。**核心价值：避免下游 ADS 层每张报表都重复扫描 DWD 全量明细。**

统计周期后缀说明：
- `_1d`：最近 1 日（当日分区独立，每日增量运行）
- `_nd`：最近 N 日（基于 1d 表按滑动窗口聚合，含 7d 和 30d）
- `_td`：历史至今（逐日累积，当日 = 昨日 td + 当日 1d）

### 5.1 dws_trade_user_sku_order_1d（用户商品粒度下单 1 日汇总）

**一行代表：** 一个用户对一个 SKU 当日的下单汇总。

**聚合维度：** user_id + sku_id + dt

**度量值：** 下单次数(COUNT)、下单件数(SUM sku_num)、原始金额、活动优惠、优惠券优惠、实付金额

**维度退化：** JOIN dim_sku_full 冗余 sku_name、三级品类路径、品牌，避免下游反复 JOIN 维度表。

### 5.2 dws_trade_user_order_1d（用户粒度下单 1 日汇总）

**一行代表：** 一个用户当日的下单汇总（去重订单数）。

**聚合维度：** user_id + dt

**与 5.1 的区别：** 去掉了 SKU 粒度，同一个用户当日多笔订单聚合为一行。`order_count_1d = COUNT(DISTINCT order_id)`。

### 5.3 dws_trade_user_cart_add_1d（用户粒度加购 1 日汇总）

**一行代表：** 一个用户当日的加购汇总。

**度量值：** 加购次数、加购件数

### 5.4 dws_trade_user_payment_1d（用户粒度支付 1 日汇总）

**一行代表：** 一个用户当日的支付成功汇总。

**度量值：** 支付次数、支付件数、支付金额

### 5.5 dws_trade_province_order_1d（省份粒度下单 1 日汇总）

**一行代表：** 一个省份当日的下单汇总。

**维度退化：** JOIN dim_province_full 冗余 province_name、area_code、iso_code、iso_3166_2、region_name

**业务作用：** 中国地图可视化（Superset 需要 ISO 编码做地理映射）。

### 5.6 dws_traffic_session_page_view_1d（会话粒度浏览 1 日汇总）

**一行代表：** 一个用户会话当日的页面浏览汇总。

**度量值：** 访问时长(SUM during_time)、浏览页面数(COUNT)

**业务作用：** 统计各渠道的会话数、平均停留时长、平均浏览页面数、跳出率（只有一个页面的会话比例）。

### 5.7 dws_traffic_page_visitor_page_view_1d（访客页面粒度浏览 1 日汇总）

**一行代表：** 一个设备对一个页面当日的浏览汇总。

**业务作用：** 统计每个页面的 UV、浏览时长，用户行为漏斗中"首页浏览人数"和"商品详情页浏览人数"的来源。

### 5.8 dws_tool_coupon_used_1d（优惠券使用 1 日汇总）

**一行代表：** 一张优惠券当日的使用汇总。

**度量值：** 使用次数

### 5.9 dws_interaction_sku_favor_add_1d（商品收藏 1 日汇总）

**一行代表：** 一个 SKU 当日的被收藏汇总。

**度量值：** 收藏次数

### 5.10 dws_trade_user_sku_order_nd（用户商品粒度下单 N 日汇总）

**一行代表：** 一个用户对一个 SKU 最近 7 日和 30 日的下单汇总。

**计算逻辑：** 从 5.1 的 1 日表聚合：`SUM(IF(dt >= date_add(基准日, -6), 指标, 0))` 得 7 日值，同理得 30 日值。

**业务作用：** 品牌复购率（通过 30 日下单次数 ≥2 判定）、品牌/品类中长期下单统计。

### 5.11 dws_trade_user_order_td（用户粒度下单历史至今汇总）

**一行代表：** 一个用户从注册到昨日的累计下单汇总。

**计算逻辑：** 首日 = 全量聚合；每日 = 昨日 td FULL OUTER JOIN 今日 1d，各度量值累加。

**关键字段：** order_date_first（首次下单日期，用于判定新增下单用户）、order_date_last（末次下单日期）、累计次数/金额

**业务作用：** 新增下单用户统计（order_date_first 在统计周期内）、用户生命周期分析。

### 5.12 dws_user_user_login_td（用户粒度登录历史至今汇总）

**一行代表：** 一个用户从注册到昨日的登录汇总。

**关键字段：** login_date_first（首次登录日期 ≈ 注册日期）、login_date_last（末次登录日期）、login_count_td（累计登录次数）

**业务作用：** 用户留存分析（login_date_first 判定新增，login_date_last 判定留存/流失/回流）、新增/活跃用户统计。

## 第六部分：ADS 层（16 张应用报表）

ADS 层直接面向具体业务需求，每一张表对应 Superset 上的一个图表或 MySQL 中的一张报表。采用 TextFile 格式便于 DataX 导出。

### 流量主题（2 张）

| 表名 | 业务问题 | 数据来源 |
|------|---------|---------|
| `ads_traffic_stats_by_channel` | 各渠道每天/7天/30天来了多少人？平均看几个页面？跳出率多少？ | dws_traffic_session_page_view_1d（按 channel 聚合） |
| `ads_page_path` | 用户在 App 里是怎么跳转的？从首页→详情页→下单页的转化路径是什么？ | dwd_traffic_page_view_inc（LEAD 函数取下一页） |

### 用户主题（6 张）

| 表名 | 业务问题 | 计算逻辑 |
|------|---------|---------|
| `ads_user_change` | 今天流失了多少用户？回流了多少？ | 流失：login_date_last = 7天前；回流：昨日流失 + 今日登录 |
| `ads_user_retention` | 7 天前注册的用户，到今天还剩多少活跃？ | login_date_first=新增日，login_date_last=今日→留存；按首次登录日分组，留存数÷新增数 |
| `ads_user_stats` | 最近 1/7/30 天新增多少用户？活跃多少？ | 新增：login_date_first 在窗口内；活跃：login_date_last 在窗口内 |
| `ads_user_action` | 从浏览首页到支付的漏斗各阶段转化率是多少？ | 首页浏览→商品详情→加购→下单→支付，每阶段人数÷上一阶段人数 |
| `ads_new_order_user_stats` | 最近 1/7/30 天有多少新用户首次下单？ | dws_trade_user_order_td，order_date_first 在窗口内 |
| `ads_order_continuously_user_count` | 最近 7 天有多少用户连续 3 天下单？（用户黏性指标） | LEAD(dt, 2) OVER user_id，datediff=2 |

### 商品主题（5 张）

| 表名 | 业务问题 | 计算逻辑 |
|------|---------|---------|
| `ads_repeat_purchase_by_tm` | 各品牌 30 天复购率是多少？ | 同一用户对一个品牌 30 天下单 ≥2 次 / 下单 ≥1 次 |
| `ads_order_stats_by_tm` | 各品牌 1/7/30 天下单数和下单人数？ | 从 1d 表(1日) + nd 表(7/30日) 合并 |
| `ads_order_stats_by_cate` | 各品类 1/7/30 天下单数和下单人数？ | 同上，按三级品类分组 |
| `ads_sku_cart_num_top3_by_cate` | 每个三级品类下购物车存量最多的 3 个 SKU？ | dwd_trade_cart_full 按品类 RANK 取前 3 |
| `ads_sku_favor_count_top3_by_tm` | 每个品牌下被收藏最多的 3 个 SKU？ | dws_interaction_sku_favor_add_1d 按品牌 RANK 取前 3 |

### 交易主题（2 张）

| 表名 | 业务问题 | 计算逻辑 |
|------|---------|---------|
| `ads_order_to_pay_interval_avg` | 用户平均下单后多久完成支付？（衡量支付体验） | dwd_trade_trade_flow_acc 中 AVG(unix_timestamp(payment_time) - unix_timestamp(order_time)) |
| `ads_order_by_province` | 各省份 1/7/30 天产生多少订单和金额？（中国地图热力图） | dws_trade_province_order_1d 当日 + 近 N 日聚合 |

### 优惠券主题（1 张）

| 表名 | 业务问题 | 计算逻辑 |
|------|---------|---------|
| `ads_coupon_stats` | 每种优惠券今天被用了多少次？多少人用了？ | dwd_tool_coupon_used_inc 按 coupon_id 统计次数和去重用户数 |

## 第七部分：核心业务指标速查

| 指标 | 定义 | 计算来源 |
|------|------|---------|
| **GMV** | 支付成功总金额 | dws_trade_user_payment_1d → SUM(payment_amount_1d) |
| **下单金额** | 下单总金额 | dws_trade_user_order_1d → SUM(order_total_amount_1d) |
| **UV** | 独立访客数 | dws_traffic_session_page_view_1d → COUNT(DISTINCT mid_id) |
| **PV** | 页面浏览数 | dws_traffic_session_page_view_1d → SUM(page_count_1d) |
| **转化率** | 下单人数 / 访客人数 | ads_user_action 中各阶段比值 |
| **复购率** | 下单 ≥2 次的人数 / 下单 ≥1 次的人数 | ads_repeat_purchase_by_tm |
| **留存率** | 第 N 天仍活跃的注册用户 / 当天注册用户 | ads_user_retention |
| **跳出率** | 只有一个页面的会话数 / 总会话数 | ads_traffic_stats_by_channel |
| **DAU** | 日活跃用户数 | ads_user_stats WHERE recent_days=1 → active_user_count |
| **下单到支付耗时** | 支付时间 - 下单时间的平均秒数 | ads_order_to_pay_interval_avg |
